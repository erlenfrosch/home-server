# k3s-Referenz

Dieses Dokument behandelt k3s-Konfiguration, häufige Operationen und Wartung.

---

## Konfigurationsdatei

Die k3s-Server-Konfiguration liegt auf dem Server unter `/etc/rancher/k3s/config.yaml`.

**Pfad:** `/etc/rancher/k3s/config.yaml`

```yaml
# k3s server configuration
write-kubeconfig-mode: "0644"
tls-san:
  - "192.168.1.100"    # Server-LAN-IP
  - "homeserver"       # Hostname
  - "127.0.0.1"        # Loopback
cluster-cidr: "10.42.0.0/16"    # Pod-Netz-CIDR
service-cidr: "10.43.0.0/16"    # Service-Netz-CIDR
node-name: "homeserver"
```

Nach Änderung k3s neu starten:

```bash
sudo systemctl restart k3s
```

**Service-Unit:** `/etc/systemd/system/k3s.service`

**Datenverzeichnis:** `/var/lib/rancher/k3s/`

**Kubeconfig:** `/etc/rancher/k3s/k3s.yaml` (root) und `/home/ubuntu/.kube/config` (ubuntu)

## kubectl-Cheatsheet

### Cluster-Status

```bash
# Node-Status
kubectl get nodes
kubectl get nodes -o wide              # mit IPs und OS-Info

# Alle Pods aus allen Namespaces
kubectl get pods -A
kubectl get pods -A -o wide            # mit Node und IP

# Alle Resources in einem Namespace
kubectl get all -n argocd

# Cluster-Info
kubectl cluster-info

# Verfügbare API-Resources
kubectl api-resources
```

### Workload-Operationen

```bash
# Resource beschreiben (Events, Config)
kubectl describe pod <pod-name> -n <namespace>
kubectl describe node homeserver

# Logs
kubectl logs <pod-name> -n <namespace>
kubectl logs <pod-name> -n <namespace> --previous    # vorher abgestürzter Container
kubectl logs <pod-name> -n <namespace> -f            # follow
kubectl logs <pod-name> -n <namespace> -c <container> # spezifischer Container

# Kommando im Pod ausführen
kubectl exec -it <pod-name> -n <namespace> -- /bin/sh

# Port-Forward
kubectl port-forward svc/<service-name> 8080:80 -n <namespace>
kubectl port-forward pod/<pod-name> 8080:80 -n <namespace>

# Deployment skalieren
kubectl scale deployment <name> --replicas=2 -n <namespace>

# Deployment neu starten (Rolling-Restart)
kubectl rollout restart deployment/<name> -n <namespace>

# Rollout-Status
kubectl rollout status deployment/<name> -n <namespace>

# Rollout-Historie
kubectl rollout history deployment/<name> -n <namespace>

# Rollback
kubectl rollout undo deployment/<name> -n <namespace>
```

### Resource-Inspektion

```bash
# YAML einer Resource
kubectl get deployment <name> -n <namespace> -o yaml

# Secret-Wert (base64-decoded)

```

### Namespace-Operationen

```bash
# Namespaces auflisten
kubectl get namespaces

# Namespace erstellen
kubectl create namespace my-app

# Namespace löschen (löscht alle Resources darin!)
kubectl delete namespace my-app

# Default-Namespace für aktuellen Context setzen
kubectl config set-context --current --namespace=argocd
```

### Apply & Delete

```bash
# Manifest applyen
kubectl apply -f manifest.yaml
kubectl apply -f directory/            # alle YAMLs im Verzeichnis

# Resource löschen
kubectl delete -f manifest.yaml
kubectl delete pod <name> -n <namespace>
kubectl delete pod <name> -n <namespace> --force --grace-period=0  # Force-Kill

# Dry-Run (Vorschau)
kubectl apply -f manifest.yaml --dry-run=client
kubectl apply -f manifest.yaml --dry-run=server
```

## Storage — local-path-Provisioner

k3s liefert die `local-path`-StorageClass standardmäßig mit. Sie provisioniert
PersistentVolumes über Host-Path-Verzeichnisse auf der NVMe-SSD.

```bash
# StorageClass prüfen
kubectl get storageclass

# Erwartet:
# NAME                   PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE
# local-path (default)   rancher.io/local-path   Delete          WaitForFirstConsumer
```

**Daten-Speicherort auf dem Host:** `/var/lib/rancher/k3s/storage/`

**Beispiel-PVC:**

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-data
  namespace: my-app
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: local-path
  resources:
    requests:
      storage: 10Gi
```

**Wichtige Limitierungen:**

- Nur `ReadWriteOnce` — nicht parallel von mehreren Pods mountbar.
- Daten leben auf dem einzigen Node — nicht repliziert.
- `Delete`-Reclaim-Policy — beim Löschen der PVC werden die PV-Daten ebenfalls gelöscht.
- `WaitForFirstConsumer` — PV wird erst angelegt, wenn ein Pod gescheduled wird.

## Networking — Flannel VXLAN

k3s nutzt Flannel im VXLAN-Mode für Pod-zu-Pod-Networking.

```bash
# Flannel-Pods
kubectl get pods -n kube-system -l app=flannel

# CNI-Konfiguration
cat /var/lib/rancher/k3s/agent/etc/cni/net.d/10-flannel.conflist

# Pod-Netz-CIDR
kubectl get nodes -o jsonpath='{.items[*].spec.podCIDR}'
```

## Ingress — Traefik

k3s deployt Traefik v2 als Default-Ingress-Controller und bindet die Host-Ports 80 und 443.

```bash
# Traefik-Pods
kubectl get pods -n kube-system -l app.kubernetes.io/name=traefik

# Traefik-Service (zeigt NodePort-Mapping)
kubectl get svc -n kube-system traefik

# Traefik-Dashboard (aus Sicherheitsgründen standardmäßig deaktiviert)
kubectl port-forward -n kube-system svc/traefik 9000:9000
# Dann http://localhost:9000/dashboard/ öffnen
```

**Standard-Kubernetes-Ingress (Beispiel):**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app
  namespace: my-app
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: web
spec:
  ingressClassName: traefik
  rules:
    - host: my-app.homeserver
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-app-svc
                port:
                  number: 80
```

**Traefik-IngressRoute (native CRD):**

```yaml
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: my-app
  namespace: my-app
spec:
  entryPoints:
    - web
  routes:
    - match: Host(`my-app.homeserver`)
      kind: Rule
      services:
        - name: my-app-svc
          port: 80
```

## Helm-Integration

Helm wird vom Ansible-Playbook unter `/usr/local/bin/helm` installiert.

```bash
# Helm verifizieren
helm version

# Installierte Releases auflisten
helm list -A

# Release-Values anzeigen
helm get values argocd -n argocd

# Release upgraden
helm upgrade argocd argo/argo-cd -n argocd -f /tmp/my-values.yaml

# Release deinstallieren
helm uninstall argocd -n argocd

# Charts suchen
helm search repo traefik
helm search hub prometheus

# Repo hinzufügen
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
```

## k3s upgraden

Zum Upgrade `k3s_version` in `ansible/group_vars/all.yml` setzen und das Playbook neu laufen lassen:

```bash
# In ansible/group_vars/all.yml
k3s_version: v1.30.1+k3s1

# Nur die k3s-Rolle ausführen
make k3s
# oder:
ansible-playbook -i ansible/inventory/hosts.yml ansible/site.yml --tags k3s --ask-vault-pass
```

Alternativ manuell auf dem Server:

```bash
# Aktuelle Version
k3s --version

# Upgrade via Install-Skript (idempotent, upgraded bei abweichender Version)
curl -sfL https://get.k3s.io | INSTALL_K3S_VERSION=v1.30.1+k3s1 sh -s - server

# Verifizieren
kubectl get nodes
k3s --version
```

**Vor dem Upgrade:**

- [k3s-Releases](https://github.com/k3s-io/k3s/releases) auf Breaking Changes prüfen.
- [Kubernetes-Changelog](https://kubernetes.io/releases/) auf API-Deprecations prüfen.
- `/etc/rancher/k3s/` und `/var/lib/rancher/k3s/` sichern.

## Backup & Restore

k3s nutzt SQLite (single-node) als Datastore.

```bash
# SQLite-DB sichern
sudo cp /var/lib/rancher/k3s/server/db/state.db ~/k3s-backup-$(date +%Y%m%d).db

# Komplettes k3s-Config + Certs sichern
sudo tar -czf ~/k3s-backup-$(date +%Y%m%d).tar.gz \
  /etc/rancher/k3s/ \
  /var/lib/rancher/k3s/server/

# Alle YAML-Manifests exportieren (GitOps-Alternative)
kubectl get all -A -o yaml > ~/k8s-resources-backup-$(date +%Y%m%d).yaml
```

Für kritische Daten ist GitOps der bevorzugte Weg — alle Application-Definitionen
liegen im Git-Repo, ein erneuter Playbook-Run + ArgoCD-Sync stellt den Cluster
in den gewünschten Zustand zurück.
