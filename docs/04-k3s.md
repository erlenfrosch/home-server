# k3s Configuration Reference

This document covers k3s configuration, common operations, and maintenance tasks.

---

## Configuration File

k3s server configuration is stored at `/etc/rancher/k3s/config.yaml` on the server.

**Location:** `/etc/rancher/k3s/config.yaml`

```yaml
# k3s server configuration
write-kubeconfig-mode: "0644"
tls-san:
  - "192.168.1.100"    # Server LAN IP
  - "homeserver"       # Hostname
  - "127.0.0.1"        # Loopback
cluster-cidr: "10.42.0.0/16"    # Pod network CIDR
service-cidr: "10.43.0.0/16"    # Service network CIDR
node-name: "homeserver"
```

After changing this file, restart k3s:

```bash
sudo systemctl restart k3s
```

**Service unit:** `/etc/systemd/system/k3s.service`

**Data directory:** `/var/lib/rancher/k3s/`

**Kubeconfig:** `/etc/rancher/k3s/k3s.yaml` (root) and `/home/ubuntu/.kube/config` (ubuntu user)

### Workload Operations

```bash
# Describe a resource (events, config)
kubectl describe pod <pod-name> -n <namespace>
kubectl describe node homeserver

# Logs
kubectl logs <pod-name> -n <namespace>
kubectl logs <pod-name> -n <namespace> --previous    # crashed container
kubectl logs <pod-name> -n <namespace> -f            # follow
kubectl logs <pod-name> -n <namespace> -c <container> # specific container

# Execute command in pod
kubectl exec -it <pod-name> -n <namespace> -- /bin/sh

# Port forward
kubectl port-forward svc/<service-name> 8080:80 -n <namespace>
kubectl port-forward pod/<pod-name> 8080:80 -n <namespace>

# Scale deployment
kubectl scale deployment <name> --replicas=2 -n <namespace>

# Restart deployment (rolling restart)
kubectl rollout restart deployment/<name> -n <namespace>

# Rollout status
kubectl rollout status deployment/<name> -n <namespace>

# Rollout history
kubectl rollout history deployment/<name> -n <namespace>

# Rollback
kubectl rollout undo deployment/<name> -n <namespace>
```


## Storage — local-path Provisioner

k3s includes the `local-path` StorageClass by default. It provisions PersistentVolumes using host-path directories on the NVMe SSD.

```bash
# Check StorageClass
kubectl get storageclass

# Expected output:
# NAME                   PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE
# local-path (default)   rancher.io/local-path   Delete          WaitForFirstConsumer
```

**Data location on host:** `/var/lib/rancher/k3s/storage/`

**Example PVC:**

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-data
  namespace: my-app
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: local-path
  resources:
    requests:
      storage: 10Gi
```

**Important limitations:**
- `ReadWriteOnce` only — cannot be mounted by multiple pods simultaneously
- Data stays on the single node — not replicated
- `Delete` reclaim policy — PV data is deleted when PVC is deleted
- `WaitForFirstConsumer` binding mode — PV is created only when a pod is scheduled

## Networking — Flannel VXLAN

k3s uses Flannel in VXLAN mode for pod-to-pod networking.

```bash
# Check Flannel pods
kubectl get pods -n kube-system -l app=flannel

# Check CNI configuration
cat /var/lib/rancher/k3s/agent/etc/cni/net.d/10-flannel.conflist

# Pod network CIDR
kubectl get nodes -o jsonpath='{.items[*].spec.podCIDR}'
```

**Key network ranges:**

| Range           | Purpose              |
|-----------------|----------------------|
| 10.42.0.0/16   | Pod IPs (Flannel)    |
| 10.43.0.0/16   | Service ClusterIPs   |
| 10.43.0.10      | CoreDNS ClusterIP    |

## Ingress — Traefik

k3s deploys Traefik v2 as the default Ingress controller, listening on host ports 80 and 443.

```bash
# Traefik pods
kubectl get pods -n kube-system -l app.kubernetes.io/name=traefik

# Traefik service (shows NodePort mapping)
kubectl get svc -n kube-system traefik

# Traefik dashboard (disabled by default for security)
kubectl port-forward -n kube-system svc/traefik 9000:9000
# Then open http://localhost:9000/dashboard/
```

**Standard Kubernetes Ingress example:**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app
  namespace: my-app
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: web
spec:
  ingressClassName: traefik
  rules:
    - host: my-app.homeserver
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-app-svc
                port:
                  number: 80
```

**Traefik IngressRoute (native CRD) example:**

```yaml
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: my-app
  namespace: my-app
spec:
  entryPoints:
    - web
  routes:
    - match: Host(`my-app.homeserver`)
      kind: Rule
      services:
        - name: my-app-svc
          port: 80
```

## Upgrading k3s

To upgrade k3s to a newer version, update `k3s_version` in `ansible/group_vars/all.yml` and re-run the playbook:

```bash
# In ansible/group_vars/all.yml
k3s_version: v1.30.1+k3s1

# Run only the k3s role
ansible-playbook -i ansible/inventory/hosts.yml ansible/site.yml --tags k3s --ask-vault-pass
```

Or upgrade manually on the server:

```bash
# Check current version
k3s --version

# Upgrade via install script (idempotent, upgrades if version differs)
curl -sfL https://get.k3s.io | INSTALL_K3S_VERSION=v1.30.1+k3s1 sh -s - server

# Verify
kubectl get nodes
k3s --version
```

**Before upgrading:**
- Check the [k3s releases](https://github.com/k3s-io/k3s/releases) for breaking changes
- Check the [Kubernetes changelog](https://kubernetes.io/releases/) for API deprecations
- Back up `/etc/rancher/k3s/` and `/var/lib/rancher/k3s/`

## Backup and Restore

k3s uses SQLite (single-node) for the datastore.

```bash
# Backup SQLite database
sudo cp /var/lib/rancher/k3s/server/db/state.db ~/k3s-backup-$(date +%Y%m%d).db

# Backup full k3s config and certificates
sudo tar -czf ~/k3s-backup-$(date +%Y%m%d).tar.gz \
  /etc/rancher/k3s/ \
  /var/lib/rancher/k3s/server/

# Export all YAML manifests (GitOps alternative to backup)
kubectl get all -A -o yaml > ~/k8s-resources-backup-$(date +%Y%m%d).yaml
```

For critical data, the recommended approach is GitOps — all application definitions are in Git, so re-running the playbook restores the cluster to the desired state.
