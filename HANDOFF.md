## Offener Bug: scanbd startet nicht — Scan-Button reagiert nicht

**Symptom:** Hardware-Button am Fujitsu ScanSnap iX500 löst keinen Scan aus. `pgrep -a scanbd` liefert leeres Ergebnis.

**Bisherige Fixes (alle gemergt):**
| PR | Fix |
|----|-----|
| #22 | SMB-Share-Pfad auf IP statt Hostname; User `jaydee` statt `paperless` |
| #23+#24 | Jinja2-Konflikt `${#pages[@]}` → `{% raw %}page_count=${#pages[@]}{% endraw %}` |
| #25 | ImageMagick-Policy-Pfad: `-6` → `-7` (Ubuntu 26.04 hat IM7) |
| #27 | `PrivateTmp=true` entfernt (konfliktiert mit `ReadWritePaths=/var/tmp/scanner`) |
| #28 | `meta: flush_handlers` + `daemon_reload: true` vor `state: started` |
| #29 | `saned {` → `saned = {` (libconfig Named-Group-Syntax) |
| #30 | `saned = { user/group }` → `saned = "/usr/sbin/saned"` (String, kein Block!) |

**Aktueller Stand:** PR #30 ist gemergt, aber die neue Config wurde noch **nicht auf den Server deployed** (Ansible wurde unterbrochen — braucht interaktiven Vault-Pass).

**Nächster Schritt:**
```bash
# Vault-Pass-Datei anlegen (einmalig):
echo 'DEIN_VAULT_PASSWORT' > ~/.vault_pass && chmod 600 ~/.vault_pass

# Scanner-Role deployen:
ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_pass make scanner
# oder direkt:
ansible-playbook -i ansible/inventory/hosts.yml ansible/site.yml \
  --tags scanner --vault-password-file ~/.vault_pass
```

Alternativ: Config direkt auf dem Server testen (kein Vault nötig):
```bash
ssh -i ~/.ssh/id_ed25519 erlenfrosch@192.168.1.109

# Config-Syntax prüfen:
sudo scanbd -d 7 2>&1 | head -30

# Falls OK — Service starten:
sudo systemctl daemon-reload && sudo systemctl restart scanbd
sudo systemctl status scanbd

# Button drücken, dann Log prüfen:
sudo journalctl -u scanbd -f
```

**Falls scanbd startet aber Button nicht reagiert:**
- `sudo scanbd -d 7` zeigt erkannte SANE-Optionen des Fujitsu
- Filter `^scan.*` prüfen — Fujitsu iX500 exponiert den Button ggf. unter anderem Namen
- Referenz: `/etc/scanbd/scanner.d/fujitsu.conf` (Package-Referenz ohne scan-Action!)
